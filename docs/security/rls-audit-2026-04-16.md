## 3. Severity-Ranked Bug List

### CRITICAL

**C1. `user_profiles` — privilege escalation**

Any authenticated user can UPDATE `user_profiles` and set `role = 'owner'` for their own row. This bypasses every frontend role check. An agent opening browser devtools can become an owner in one Supabase query:

```sql
-- From browser devtools with the anon key:
UPDATE user_profiles SET role = 'owner' WHERE id = auth.uid();
```

Note: the `sync_user_profile_role` trigger only fires on INSERT or UPDATE OF `employee_id`, so a direct role update is NOT overwritten by the trigger.

**C2. `employees` — full salary exposure**

Any authenticated user can `SELECT monthly_base_salary, daily_discount_rate, kpi_bonus_amount FROM employees`. The permission model says agents should see "own only" for pay info, team leads should see "no" pay for other employees.

**C3. `payroll_records` — full payroll exposure**

Any authenticated user can read `calculated_net_pay`, `days_absent`, `kpi_achieved`, `additional_bonuses`, `overrides_json` for every employee.

**C4. `invoice_lines` — billing rate exposure**

`unit_price` and `total_price` per agent are visible to all authenticated users. This reveals what each agent is billed to clients at.

### HIGH

**H1. `time_clock` — attendance falsification**

Any authenticated user can UPDATE/DELETE time_clock entries for any employee. An agent could erase late arrivals or fabricate clock-in times.

**H2. `time_off_requests` — self-approval**

Any authenticated user can UPDATE the `status` and `reviewed_by` fields on any request, including their own.

**H3. `eod_logs` — metric tampering**

Any authenticated user can UPDATE EOD metrics for any employee — inflating KPI numbers or hiding poor performance.

**H4. `invoices` — invoice tampering**

Any authenticated user can create, modify, or delete invoices and change their status.

**H5. Role check depends on `user_profiles.role` being populated**

The app reads `user_profiles.role` for permission decisions. If a user signs up but the `sync_user_profile_role` trigger hasn't fired (e.g. `employee_id` is NULL), the role may be stale or manually set. The trigger only fires on INSERT or `UPDATE OF employee_id`, meaning a direct `UPDATE user_profiles SET role = ...` is permanent.

### MEDIUM

**M1. `payroll_periods` — period manipulation**

Agents can create or close payroll periods, which could disrupt payroll runs.

**M2. `clients` / `campaigns` / `campaign_kpi_config` / `shift_settings` — config tampering**

Agents can modify operational configuration tables. No PII exposure but could disrupt operations.

### LOW

**L1. Duplicate overlapping policies on early tables**

`employees`, `payroll_periods`, `payroll_records`, `clients`, `invoices`, `invoice_lines` each have TWO `FOR ALL` policies from different migrations (the early `USING (true)` policies from `20260401165427`/`20260401230853` and the `auth.role() = 'authenticated'` policies from `full_setup`). These are functionally equivalent but create confusion. The early `USING (true)` policies are slightly more permissive in theory (they don't check `auth.role()`), though in practice Supabase's `authenticated` role is applied by the JWT.

**L2. `full_setup` uses `auth.role() = 'authenticated'` instead of `TO authenticated`**

The `full_setup` policies use `USING (auth.role() = 'authenticated')` in the USING clause rather than the more idiomatic `FOR ALL TO authenticated USING (true)`. Functionally equivalent but less readable.

## 4. Proposed Fix SQL

All fixes below are **proposed only — not applied**.

### Fix C1: Lock down `user_profiles`

```sql
-- Drop the blanket policy
DROP POLICY "Allow full access for authenticated" ON public.user_profiles;

-- Users can read their own profile
CREATE POLICY "Users read own profile"
  ON public.user_profiles FOR SELECT TO authenticated
  USING (id = auth.uid());

-- Leadership can read all profiles (for user management UI)
CREATE POLICY "Leadership read all profiles"
  ON public.user_profiles FOR SELECT TO authenticated
  USING (
    EXISTS (
      SELECT 1 FROM public.user_profiles up
      JOIN public.employees e ON up.employee_id = e.id
      WHERE up.id = auth.uid()
        AND e.title IN ('owner', 'admin', 'manager')
    )
  );

-- Only the sync trigger (SECURITY DEFINER) should write roles.
-- Users can insert their own profile on signup:
CREATE POLICY "Users insert own profile"
  ON public.user_profiles FOR INSERT TO authenticated
  WITH CHECK (id = auth.uid());

-- No direct UPDATE/DELETE for anyone via RLS.
-- Role changes go through employees.title → trigger sync.
```

### Fix C2: Lock down `employees`

```sql
-- Drop all existing blanket policies
DROP POLICY IF EXISTS "Allow full access to employees" ON public.employees;
DROP POLICY IF EXISTS "Allow full access for authenticated" ON public.employees;

-- Leadership sees everything
CREATE POLICY "Leadership full read on employees"
  ON public.employees FOR SELECT TO authenticated
  USING (
    EXISTS (
      SELECT 1 FROM public.user_profiles up
      JOIN public.employees e ON up.employee_id = e.id
      WHERE up.id = auth.uid()
        AND e.title IN ('owner', 'admin', 'manager')
    )
  );

-- Team leads see employees in their campaigns (but a view/function
-- should strip salary columns — RLS can't do column-level filtering).
-- For now, team leads see the row but the frontend must hide pay fields.
-- IMPORTANT: See Open Question Q1 about column-level pay hiding.
CREATE POLICY "Team leads read own campaign employees"
  ON public.employees FOR SELECT TO authenticated
  USING (
    EXISTS (
      SELECT 1 FROM public.user_profiles up
      JOIN public.employees tl ON up.employee_id = tl.id
      WHERE up.id = auth.uid()
        AND tl.title = 'team_lead'
        AND employees.campaign_id IN (
          SELECT c.id FROM public.campaigns c WHERE c.team_lead_id = tl.id
        )
    )
  );

-- Agents see only their own row
CREATE POLICY "Agents read own employee row"
  ON public.employees FOR SELECT TO authenticated
  USING (
    EXISTS (
      SELECT 1 FROM public.user_profiles up
      WHERE up.id = auth.uid()
        AND up.employee_id = employees.id
    )
  );

-- Only leadership can INSERT/UPDATE/DELETE employees
CREATE POLICY "Leadership write employees"
  ON public.employees FOR ALL TO authenticated
  USING (
    EXISTS (
      SELECT 1 FROM public.user_profiles up
      JOIN public.employees e ON up.employee_id = e.id
      WHERE up.id = auth.uid()
        AND e.title IN ('owner', 'admin', 'manager')
    )
  )
  WITH CHECK (
    EXISTS (
      SELECT 1 FROM public.user_profiles up
      JOIN public.employees e ON up.employee_id = e.id
      WHERE up.id = auth.uid()
        AND e.title IN ('owner', 'admin', 'manager')
    )
  );
```

## 5. Things NOT Covered by RLS

### 5.1 SECURITY DEFINER functions

Three functions use `SECURITY DEFINER`, meaning they bypass RLS:

1. **`auto_clockout_overdue()`** (`20260413120100`) — Runs via pg_cron every 5 minutes. Updates `time_clock` rows. Safe: it's a system job, not user-invocable via the API. However, it has no explicit `REVOKE EXECUTE` from `anon`/`authenticated`, so technically a client could call `SELECT public.auto_clockout_overdue()` and trigger early clock-outs.

2. **`sync_user_profile_role()`** (`20260413140000`) — Trigger function that copies `employees.title` into `user_profiles.role`. Safe as a trigger, but because it's `SECURITY DEFINER` it reads `employees` bypassing RLS. This is intentional.

3. **`log_shift_settings_change()`** (`20260413140000`) — Trigger that inserts into `shift_settings_audit`. Safe: writes audit rows that the trigger's invoker couldn't write directly (audit table is SELECT-only).

**Risk:** `auto_clockout_overdue()` is callable by any authenticated user unless explicitly revoked. An agent could invoke it to force early clock-outs on open shifts that have passed their `shift_end_expected + 30min`.

### 5.2 Seed data with plaintext salaries

`20260413000003_update_salaries.sql` contains every employee's exact salary, bonus, and daily discount rate in plaintext SQL committed to the git repository. Even after RLS is fixed, anyone with repo access can read all compensation data. This is a separate concern from RLS but worth noting.

### 5.3 `assign_employee_id()` function

(`20260415000001`) — Not `SECURITY DEFINER`, runs as the invoking user. This is a `BEFORE INSERT` trigger on employees. Since any authenticated user currently has INSERT on employees, they could create new employee records. After the proposed RLS fixes, only leadership could trigger this.

### 5.4 Cascade triggers bypass intent

`sync_reports_to_from_campaign()` and `cascade_campaign_tl_change()` (`20260417000001`) run as the invoking user. Currently any authenticated user can UPDATE campaigns and trigger these cascades. After RLS is tightened this is fine.

## 6. Open Questions for D

**Q1. Column-level pay hiding for team leads.**
RLS is row-level only — it can't hide `monthly_base_salary` from a team lead while showing them `full_name` on the same row. Options:
- (a) Create a Postgres VIEW that omits salary columns and have team leads query the view instead of the table
- (b) Accept that team leads can technically see salaries at the DB level and rely on the frontend to not display them (weaker guarantee)
- (c) Move salary data to a separate `employee_compensation` table that only leadership can access via RLS
Which approach do you prefer?

**Q2. Should agents see campaign/client names?**
The proposed fixes allow all authenticated users to SELECT from `campaigns` and `clients`. This seems reasonable (agents need to know their campaign name), but confirm this matches your intent.

**Q3. `auto_clockout_overdue()` — revoke public execute?**
Should we add `REVOKE EXECUTE ON FUNCTION public.auto_clockout_overdue() FROM authenticated, anon;` so only pg_cron can call it?

**Q4. Team leads and time_clock INSERT/UPDATE.**
The permission model says team leads can "clock in/out" (row 5 in the permission table). Should team leads also be able to edit clock entries for their team members (e.g. correct a missed clock-out), or only their own?

**Q5. EOD log edits.**
Can agents UPDATE their own EOD logs after submission, or should it be insert-only (immutable once submitted)?

**Q6. Deysi Esperanza's title.**
Migration `20260414000003` promotes Deysi to `team_lead`, but the earlier `20260413140000` already set her. The second UPDATE is a no-op but confirms intent. Just flagging for awareness — no action needed.

## Summary

**13 of 15 tables have effectively no RLS** — every authenticated user (including the lowest-privilege agent) has full read/write access to all data including salaries, payroll, invoices, and user roles. The two correctly scoped tables (`shift_settings_audit` and `mexican_holidays`) are read-only reference data.

The single most dangerous issue is **C1: any agent can promote themselves to owner** by directly updating `user_profiles.role`, since the sync trigger doesn't fire on direct role updates.

All proposed fixes follow the pattern: check `employees.title` via `user_profiles` to determine the caller's role, then scope access accordingly. This creates a dependency on the `user_profiles` ↔ `employees` link being correct, which the existing `sync_user_profile_role` trigger maintains.
